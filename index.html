Overview

Sync Engine builds on the basic Gizinta library. Gizinta makes it easier to move data between databases using XML configuration files to map source datasets to target data models. Gizinta also has 'playlists' - all Gizinta tools have the ability to run one or more datasets at a time, and this makes it possible to update parts or all of a target database by just passing an XML file to the process to be run.

Sync Engine takes the Gizinta idea further by providing a way to execute enhanced playlists and update the target database with change-only updates if all playlists run successfully. Playlists are enhanced with a reference to a Safe Software FME process to run, along with details about the database views to be created for change-only updates.

The gseBase repository contains most of the code and the folder structure for the Sync Engine tools. Implementation projects are built and managed using Github private repositories, and they include most of the content from the gseBase project - with names like gseVT and gseUW.

Folders

The project is organized into multiple folders. The base location for these files on disk is typically C:\Apps\Gizinta\gseBase or the specific project that you are working on.

  • ETL (Extract, Transform and Load) Folder - the ETL folder contains the scripts that run the tools, along with subfolders for different parts of the Sync Engine.

Key Concepts

Sync Engine uses arcpy and FME to automate spatial data update processes. It is a general concept that could be used to automate many processes, with some special features like playlists, built in QA, and change detection that provide a sophisticated way to automate multiple processing steps. The concepts and implementation were developed based on many years building these kinds of solutions.

Currently the emphasis is on Campus Floorplans and Routing sync from CAD-GIS. In this space, the main requirement is to build and incrementally update a seamless 2D and 3D Campus database from source CAD drawings. This requires an industrial-strength solution that is capable of handling large volumes of data with minimal support beyond the maintenance of the source drawings. There are several challenges for a solution like this:

  • Coordinate transformation from paper space to geographic space
  • Build derived data - add the third dimension and additional data required for GIS applications
  • Perform change detection and change-only updates to the database
  • Changes should be written only if some fields change. For example, the LastUpdate field is updated every time a feature is written, so this field should not be included in change detection logic.
  • Support multiple processing steps, and if any of the steps do not pass QA rules or fail for any other reason then skip to the next drawing without posting any of the data from steps along the way
  • Provide logging capabilities that support debugging and problem solving for drawings and individual features
  • Provide full access to FME and ArcGIS capabilities
  • Support batch loading for thousands of drawings, along with incremental update that can run for days
  • Make sure the solution is supportable and manageable
